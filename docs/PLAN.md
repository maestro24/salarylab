## 1. 문제 정의 & 검증 근거 (2026-07-13 실조사)

- "연봉 5000 실수령액" 류 구간별 검색 수요가 상시 존재하며, **동일 pSEO 전략의 영세 사이트들이 실제 구글 상위에 랭킹 중** (signalplanner.co.kr/income/5000, matlocation.co.kr/salary/5000, fin.fran.kr) — 진입 가능한 SERP라는 실증.
- 네이버는 검색창에 임금계산기를 내장 → **네이버 트래픽은 포기, 구글 전용 게임**. 구글은 자체 계산기 미내장(니치 사이트 랭킹으로 확인).
- 경쟁 pSEO들은 전부 "정적 표" 수준 — **역계산·협상 시뮬레이터가 빈 틈**.
- 리스크: 후발 4~5번째 주자. 차별화 콘텐츠 없이 표만 찍으면 묻힘.

## 2. 핵심 원칙

1. **정확도가 생명**: 실수령액이 사람인 계산기와 유의미하게 다르면 사이트 죽음. 요율 데이터는 단일 JSON 소스, Python·JS 듀얼 엔진 교차 검증 (바로변환 패턴).
2. **경쟁자가 안 하는 것으로 승부**: 역계산("실수령 350 받으려면 연봉 얼마?")과 협상 시뮬레이터가 본체. 구간별 표 페이지는 유입 통로.
3. **단계 방출**: 파일럿 ≤300페이지 → 색인율 확인 → 확장 (pSEO 공통 원칙).
4. **정직한 고지**: "간이세액표·요율 기준 추정치, 회사 공제 항목에 따라 상이" 면책 상시 표기. YMYL 주제.
5. **연 1회 유지**: 요율·간이세액표 갱신 외 손대지 않는 구조 (방치 성장 모델).
6. **점진 개선**: 단계마다 시각·성능·느낌 +12%, 매 단계 브라우저 실검증.

## 5. 타겟 & 사용 맥락

- 이직 준비·연봉 협상 직전 직장인 (검색: "연봉 XXXX 실수령액")
- 첫 취업자 (오퍼 금액 → 월급 감 잡기)
- 역방향 수요: "월 실수령 400 만들려면" — 목표 기반 검색 (경쟁 미커버)

## 6. 계산 로직 (정확도 스펙)

```
월 실수령 = 월 급여 − (국민연금 + 건강보험 + 장기요양 + 고용보험 + 소득세 + 지방소득세)
```
- 국민연금 4.5% (기준소득월액 상·하한 적용 — 2026 고시값)
- 건강보험 (2026 요율) + 장기요양(건보료의 고시 비율)
- 고용보험 0.9%
- 소득세: **근로소득 간이세액표**(홈택스 공개 XLS → JSON 변환) 룩업, 부양가족 수 반영
- 지방소득세 = 소득세의 10%
- 옵션: 비과세 식대(월 20만), 부양가족/자녀 수
- **정확도 게이트**: 대표 연봉 20개 지점을 사람인 계산기 결과와 교차 대조하는 fixture 테스트 — 오차 발생 시 빌드 실패

## 7. 페이지 구조 (파일럿 ~290)

```
/                          인터랙티브 계산기 (연봉→실수령, 옵션 전부)
/salary/<만원>/            연봉별 정적 페이지 ×191
                           (1,000만~2억: 100만 단위 + 2천만~8천만은 50만 단위 보강)
/reverse/<만원>/           역계산 ×30 ("월 실수령 300 → 필요 연봉") 100만~1,000만, 실수요 구간
/table/                    2026 실수령액 총괄표 (헤드 키워드 "실수령액표" 타깃)
/negotiate/                연봉 협상 시뮬레이터 (인상률↔월 증가분, 인상 제안 문구 예시)
/guide/                    공제 항목 해설 (E-E-A-T·씬 콘텐츠 방지, 수제 1~2편)
```
- 연봉 페이지 구성: H1 즉답("연봉 5,000만 원 실수령액은 월 약 356만 원") + 공제 내역 분해 표 + 근처 연봉 표 + 역계산 링크 + FAQ 스키마
- 내부 링크 그물 + 고아 0 검증 (생성기 내장, baroconvert 재사용)
